## Scope

This report applies to **workspace Template Studio PDF** generated by `lib/workspace/exports/pdf.ts` (pdf-lib + Helvetica + embedded Arabic font). It is **not** the same pipeline as `backend/app/Services/DocumentTemplateRendererService.php`, which renders **Blade/HTML** for server-side documents using browser font stacks and a different layout engine.

## Issue sources (ranked)

### 1. Unconditional horizontal scale in `drawItems` (fixed)

**Symptom:** Column proportions in PDF could diverge from the on-screen table when declared column widths **already fit** inside the printable inner width.

**Cause:** PDF used `scale = tableW / totalDeclaredWidth` for **every** layout, forcing all columns to stretch or shrink to fill the full inner width even when the preview used narrower total table width (sum of user widths &lt; cap).

**Preview behavior after fix:** The items table uses **pixel** `<col>` widths and `table` width `min(sumWidths, cap)` so unused horizontal space is not distributed unless the user widens columns.

**PDF behavior after fix:** `scale` is `1` when `totalDeclaredWidth <= tableW`; scaling applies only for **overflow protection** (sum exceeds printable inner width).

**Affected:** All item columns; bilingual header band width and row separators now use `usedTableW = totalDeclaredWidth * scale` so backgrounds and rules match the drawn grid.

### 2. Fixed bilingual header band heights vs fluid CSS

**Symptom:** English and Arabic header labels overlapping or feeling tighter in PDF than in the browser.

**Cause:** PDF draws two horizontal bands with fixed pixel heights (`enHeaderH` / `arHeaderH`) while the browser uses line-height, padding, and font metrics from the CSS cascade.

**Mitigation applied:** Slightly increased band heights (19ŌåÆ20 px, 17ŌåÆ18 px) and reduced header font sizes by 1 pt in `TYPOGRAPHY` to align with preview typography.

**Residual risk:** Extremely long translated headers may still clip in PDF before the browser wraps; fixing that fully would require multi-line measurement in pdf-lib (not implemented here).

### 3. Different text stacks (font shaping)

**Symptom:** Arabic glyph joining or width differs subtly between preview (web fonts) and PDF (embedded subset).

**Cause:** Preview uses CSS `font-family` (e.g. Noto Naskh Arabic); PDF uses `shapeArabicForPdf` + a single embedded Arabic `PDFFont`. pdf-lib does not run the full HarfBuzz/OpenType pipeline of a browser.

**Affected:** Arabic seller card, Arabic column headers, Arabic cell text.

### 4. Fixed row height in PDF vs `word-break` in HTML

**Symptom:** Multi-line descriptions align differently; PDF may clip or center on a fixed `itemRowH` (28 px).

**Cause:** Browser cells grow vertically; PDF uses one row height per line.

**Affected:** `description`, `remarks`, long headers.

### 5. Backend HTML renderer (separate product surface)

**Cause:** `DocumentTemplateRendererService` builds different HTML/CSS; any ŌĆ£PDF from serverŌĆØ that uses dompdf/wkhtmltopdf/snappy would have **additional** engine-specific issues (RTL tables, font embedding). That path was **not** changed in this task.
