# User Workspace Constitution

Updated: 2026-04-23
Status: Subordinate constitutional document under the system constitution and master design hierarchy.

## 1. Workspace Authority

### 1.1 Clause IDs

- `UX-CON-001`: User Workspace shall be the primary operational workspace for company users performing business activity.
- `UX-CON-002`: User Workspace shall inherit from the system constitution, master design authority, master design v2, and standards framework v2.
- `UX-CON-003`: User Workspace shall not override or compete with the supreme governance layer.

### 1.2 Ownership

- `UX-CON-010`: User Workspace shall own operational business execution for sales, purchases, inventory, accounting, banking, VAT, reporting, and module setup within the user role.
- `UX-CON-011`: User Workspace shall own role-specific navigation, operational shell behavior, and workspace continuity for the user role.
- `UX-CON-012`: User Workspace shall own the presentation of business modules, registers, previews, details, and task flows exposed to company users.

### 1.3 Non-ownership

- `UX-CON-020`: User Workspace shall not own platform governance law.
- `UX-CON-021`: User Workspace shall not own admin-only control surfaces, support-only workflows, or agent-only commercial workflows.
- `UX-CON-022`: User Workspace shall not define accounting truth, document truth, or audit truth; those truths shall remain owned by their governing layers.

## 2. Workspace Shell Law

### 2.1 Shell behavior

- `UX-CON-030`: The user shell shall adapt to operational modules without losing route truth.
- `UX-CON-031`: The user shell shall expose sidebar, top bar, context bar, and action bar as governed shell surfaces.
- `UX-CON-032`: Shell surfaces shall support dense operational work without forcing unrelated navigation detours.

### 2.2 Role boundary

- `UX-CON-040`: User Workspace shall differ from admin, assistant, and agent workspaces by operational scope, role-specific actions, and sidebar content.
- `UX-CON-041`: Cross-role leakage shall be treated as a workspace governance defect.
- `UX-CON-042`: A user route shall not inherit admin or support meaning by layout coincidence.

## 3. Workflow Boundary

- `UX-CON-050`: User Workspace shall support register, create, edit, detail, preview, split-view, and report flows where those modules are defined by the master design hierarchy.
- `UX-CON-051`: Workflow continuity shall survive shell transitions, panel collapses, and viewport changes.
- `UX-CON-052`: A workspace surface shall not be complete by visual presence alone.
