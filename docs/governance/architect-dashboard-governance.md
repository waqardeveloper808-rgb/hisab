# Architect Dashboard Governance Definition

Updated: 2026-04-23
Status: Governance visibility and command-surface definition.

## 1. Dashboard Authority

### 1.1 Clause IDs

- `ARCH-001`: The Architect Dashboard shall replace the concept of Master Design Dashboard in governance language.
- `ARCH-002`: The Architect Dashboard shall be the governance visibility layer for the whole platform.
- `ARCH-003`: The Architect Dashboard shall not be the product law itself.

## 2. Required Surface

### 2.1 Visible governance state

- `ARCH-010`: The Architect Dashboard shall display constitutional status.
- `ARCH-011`: The Architect Dashboard shall display master design sections.
- `ARCH-012`: The Architect Dashboard shall display control-point coverage.
- `ARCH-013`: The Architect Dashboard shall display audit status.
- `ARCH-014`: The Architect Dashboard shall display nonconformities.
- `ARCH-015`: The Architect Dashboard shall display the root-cause queue.
- `ARCH-016`: The Architect Dashboard shall display the corrective-action queue.
- `ARCH-017`: The Architect Dashboard shall display retest status.
- `ARCH-018`: The Architect Dashboard shall display a compliance heatmap.
- `ARCH-019`: The Architect Dashboard shall display module compliance state.

### 2.2 Governance command surface

- `ARCH-020`: The Architect Dashboard shall support review and governance decision making.
- `ARCH-021`: The Architect Dashboard shall not be described as a casual overview page.
- `ARCH-022`: The Architect Dashboard shall not define product law, only surface it.

## 3. Naming Law

- `ARCH-030`: Governance documentation shall use Architect Dashboard terminology.
- `ARCH-031`: Deprecated "Master Design Dashboard" references shall be mapped in the file map.
- `ARCH-032`: Any retained runtime component names shall be treated as implementation detail, not governing language, until replaced.
