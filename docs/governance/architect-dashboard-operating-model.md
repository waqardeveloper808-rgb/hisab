# Architect Dashboard Operating Model

## 1. Purpose
1.1 The Architect Dashboard shall be the governance command surface of the platform.
1.2 The Architect Dashboard shall surface law, status, drift, and corrective workflow state.
1.3 The Architect Dashboard shall not be product law itself.

## 2. Required Surfaces
2.1 Constitutional sections.
2.2 Module compliance states.
2.3 Control-point coverage status.
2.4 Active nonconformities.
2.5 Severity heatmap.
2.6 RCA queue.
2.7 Corrective action queue.
2.8 Retest queue.
2.9 Closure history.
2.10 Module readiness state.
2.11 Evidence visibility.
2.12 Deprecated-source warnings.
2.13 Governance drift warnings.

## 3. Visibility Rule
3.1 The dashboard shall surface noncompliance honestly.
3.2 The dashboard shall not hide failed controls behind generic summaries.
3.3 The dashboard shall distinguish governed status from implementation optimism.

## 4. Clause IDs
4.1 Stable clauses in this document use the prefix `SYS-ARCH`.

## 5. Enforceable Clauses
- `SYS-ARCH-001`: The Architect Dashboard shall be the governance command surface.
- `SYS-ARCH-002`: The dashboard shall expose required status surfaces.
- `SYS-ARCH-003`: The dashboard shall surface noncompliance honestly.
- `SYS-ARCH-004`: The dashboard shall not hide failed controls.
